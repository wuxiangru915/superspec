# Superspec — Superspec + Superspec Dual-Star Development Workflow

Superspec and Superspec orbit the same goal like a binary star system.

```
Superspec handles WHAT  — outline, proposal, spec lifecycle, Deliver
Superspec handles HOW — technical Blueprint, planning, execution, closing
```

**Core principle: brainstorming cannot be skipped. Every change must undergo deep Blueprint (except hotfix and tweak presets).**

---

## Decision Core

Agents need only read this section for decision-making. Refer to the Reference Appendix as needed.

### Output Language Rule

Use the language of the user request that triggered this workflow as the default output language. When resuming an existing change with a clear dominant artifact language, preserve that language unless the user explicitly asks to switch.

### Automatic Phase Detection & Routing

To determine the next step in the workflow, **you MUST run `superspec run`**.

The CLI will analyze the project state and output a `SUGGESTED_ACTION`.
- If the CLI suggests `/superspec-Vision`, follow that.
- If it identifies an active change (e.g., "feature-x") and suggests `/superspec-Forge`, invoke that skill directly.
- **Do not manually calculate phases** or read `state.yaml` for routing unless the CLI tool fails.

**Resume Rules**:
- On every context resume, rerun `superspec run` to verify the current status.
- If `SUGGESTED_ACTION` matches a specific skill (Vision, Blueprint, Forge, Refine, Deliver), invoke it immediately.
- If multiple changes are active, the CLI will notify you; follow its selection or ask the user if needed.

### Phase Transitions

Flow chain: Vision → Blueprint → Forge → Refine → Deliver

**Decision points are blocking points**: whenever reaching any of the following nodes, the current `/superspec` invocation must stop, **using the current platform's available user input/confirmation mechanism to wait for the user's choice**.

Nodes requiring user participation:
1. Open phase proposal/Blueprint/tasks review and confirmation
2. Confirm Blueprint approach during brainstorming
3. Plan-ready pause choice during Forge phase, followed by workflow configuration selection
4. Decide to fix or accept deviation when Refine fails
5. Choose branch handling method for finishing-branch
6. Archive phase final confirmation before running the Deliver script
7. Encounter upgrade conditions (hotfix/tweak → full workflow)
8. Build phase scope expansion requiring redesign
9. Open phase large PRD requiring confirmation to split

### Best Practices

1. **Fused execution** — Clarification, Brainstorming, and Planning are now integrated directly into Superspec skills; no need to load external peer skills for these steps.
2. **delta spec is a living document** — Freely modify during phase 3, sync at Deliver.

3. **Handoff packages are generated by scripts** — Use `superspec-handoff.sh`
4. **Keep roadmap.md in sync** — Check off each completed task
5. **Commit frequently** — One commit per task, message reflects Blueprint intent
6. **Verify before Deliver confirmation** — Enter `/superspec-Deliver` only after `/superspec-Refine` passes
7. **Archive closure** — Blueprint doc and plan must mark `archived-with` gauge
8. **Modifying existing features** — Just Vision a new change
9. **Preset has limits** — Switch to full workflow promptly when upgrade conditions are met
